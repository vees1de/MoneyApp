# Entity: request

## Как понял сущность
`Request` описывает заявку сотрудника на обучение, в первую очередь внешний курс, с жизненным циклом и трассировкой решений.

## Что хранит
- Автор заявки и параметры курса: userId, courseLink, cost, dates, program.
- Бизнес-контекст: justification, departmentId, budgetSnapshot.
- Статус: draft/submitted/approved/rejected/scheduled/completed.
- Историю: комментарии, шаги согласования, вложения, audit trail.

## Что бы я предпринял
- Зафиксировать state machine заявки и допустимые переходы.
- Добавить optimistic-lock/version для безопасного редактирования.
- Ввести обязательность полей в зависимости от этапа.
- Связать заявку с календарным событием и сертификатом по завершении.

## Вопросы для уточнения
- Можно ли откатить заявку из `submitted` обратно в `draft`?
- Кто может закрывать заявку как `completed`?
- Нужны ли отдельные статусы для возврата на доработку?
